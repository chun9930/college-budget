# Toss + Banksalad 하이브리드 UI 설계 문서

## 1. 목표

이 프로젝트는 대학생이 지금 소비를 해도 되는지 즉시 판단할 수 있도록 돕는 소비 판단 도구이다.

디자인 목표는 다음과 같다.

- 홈과 예산 설정은 Toss처럼 미니멀하고 여백이 넓게 보이게 한다.
- 기록과 통계는 Banksalad처럼 숫자와 소비 정보가 또렷하게 읽히게 한다.
- 화면별 밀도는 달라도, 전체 서비스가 하나의 제품처럼 보이도록 공통 톤을 유지한다.
- 카드 남발을 줄이고, 핵심 숫자와 상태를 먼저 보여준다.

## 2. 핵심 원칙

### 2.1 공통 원칙

- 배경은 밝은 흰색 계열을 사용한다.
- 강조색은 신뢰감 있는 파란색을 기본으로 하고, 상태색은 제한적으로 사용한다.
- 라운드, 여백, 폰트 크기, 버튼 높이는 모든 화면에서 같은 기준을 사용한다.
- 섹션은 얇은 선과 넉넉한 간격으로 나눈다.
- 카드보다 정보 흐름이 먼저 읽히도록 구성한다.

### 2.2 밀도 균형 원칙

- 홈/설정은 정보량을 줄여 여백을 넓게 쓴다.
- 기록/통계는 수치와 리스트를 더 촘촘하게 배치하되, 답답하지 않게 줄 간격을 유지한다.
- 어느 화면에서도 한 화면에 너무 많은 기능을 넣지 않는다.
- 같은 텍스트 스타일과 같은 버튼 스타일을 공유해서 화면 간 낙차를 줄인다.

### 2.3 작업 기준

- 기존 기능 로직은 변경하지 않는다.
- UI 레이아웃만 수정한다.
- `index.css`에 공통 토큰을 먼저 정의한다.
- 카드 개수는 최소화한다.
- 핵심 숫자를 먼저 노출한다.
- 모바일 기준을 유지한다.

## 3. 화면별 변경안

### 3.1 홈

홈은 Toss형으로 가장 단순하게 구성한다.

포함할 요소:

- 오늘 사용 가능 금액
- 오늘 지출 금액
- 상태 메시지
- 경고 배너
- 지출 입력 버튼
- 예산 설정 버튼

구성 방식:

- 가장 중요한 수치를 상단에 크게 배치한다.
- 카드 개수는 최소화하고, 구획은 여백으로 나눈다.
- 경고 배너는 홈 상단의 얇은 강조 영역으로 보여준다.
- 상세 분석보다 빠른 판단을 우선한다.

작업 시 우선순위:

- 핵심 숫자 먼저 노출
- 카드 개수 최소화
- 모바일 기준 유지

### 3.2 예산 설정

예산 설정은 Toss형 여백을 유지하면서도, 설정 항목은 Banksalad처럼 명확하게 읽히게 한다.

포함할 요소:

- 기본 예산 카드
- 기능 토글 카드
- 목표 설정 카드
- 기간별 계산 카드

구성 방식:

- 기존 기능 구조는 유지한다.
- 카드의 테두리와 그림자는 약하게 한다.
- 토글과 숫자 입력은 서로 간격을 충분히 둔다.
- 목표 설정과 기간별 계산은 조건부로만 표시한다.

작업 시 우선순위:

- 여백 중심 구조로 단순화
- 입력과 토글의 시각적 밀도를 분리
- 핵심 수치가 먼저 보이도록 구성

### 3.3 지출 기록

지출 기록은 Banksalad형 정보 배치를 사용하되, 과도한 카드 분할은 피한다.

포함할 요소:

- 금액
- 카테고리
- 결제수단
- 고급 옵션 접기
- 템플릿 저장/불러오기
- 정기지출 등록

구성 방식:

- 입력은 빠르게 끝나도록 상단부터 순서대로 배치한다.
- 목록은 금액이 가장 먼저 보이게 한다.
- 상세 설명은 한 줄 안에서 끝내고, 길어지면 접는다.

작업 시 우선순위:

- 리스트 밀도는 홈보다 높게 유지
- 핵심 정보는 금액과 카테고리
- 보조 정보는 뒤로 배치

### 3.4 통계

통계는 Banksalad형으로 가장 수치 중심으로 구성한다.

포함할 요소:

- 월 지출 요약
- 카테고리 비중
- 가장 많이 쓴 항목
- 간단한 절약 힌트

구성 방식:

- 숫자와 비율을 먼저 보여준다.
- 차트는 보조 정보로 둔다.
- 요약과 상세를 분리하되, 화면이 무겁게 느껴지지 않도록 여백을 유지한다.

작업 시 우선순위:

- 숫자 중심 재배치
- 핵심 지표를 상단에 배치
- 차트는 보조 정보로 유지

### 3.5 캘린더

캘린더는 새로 추가되는 기록 탐색용 화면이다.

포함할 요소:

- 월간 캘린더 그리드
- 날짜별 지출 합계
- 날짜별 지출 상태
- 날짜 선택 시 해당 일자의 지출 목록

구성 방식:

- 캘린더는 보기 전용에 가깝게 유지한다.
- 날짜를 누르면 상세가 아래에 펼쳐진다.
- 편집보다 탐색을 우선한다.
- 홈과 같은 톤을 유지하되, 기록 탐색이 편하도록 정보 밀도는 조금 높인다.

작업 시 우선순위:

- 기록 탐색이 최우선
- 날짜별 지출 합계 확인
- 모바일에서도 월간 그리드가 유지되도록 구성

### 3.6 로그인 / 회원가입 / 마이페이지

- 로그인과 회원가입은 mock auth 흐름으로 단순하게 유지한다.
- 마이페이지는 사용자 정보와 설정 요약 중심으로 보여준다.
- 전체 서비스의 톤을 해치지 않도록 장식보다 정보 전달을 우선한다.

## 4. 네비게이션

- 상단 헤더는 서비스명과 핵심 이동만 보여준다.
- 하단 네비게이션은 모바일에서 홈, 예산, 기록, 통계, 캘린더 이동을 쉽게 한다.
- 로그인 관련 화면은 보조 경로로 유지한다.

## 5. 접근성 및 반응형

- 버튼 최소 높이는 48px로 유지한다.
- 본문 글자는 최소 16px 이상으로 유지한다.
- 상태는 색상과 텍스트를 함께 표시한다.
- 모바일에서는 여백을 줄이되, 카드와 리스트가 숨 막히지 않게 유지한다.
- 데스크톱에서는 정보가 너무 넓게 퍼지지 않도록 최대 폭을 제한한다.

## 6. 비목표

- 이번 작업에서 기능 로직은 바꾸지 않는다.
- 디자인만 바꾸되, 소비 판단이라는 프로젝트 목적은 유지한다.
- 캘린더는 편집 기능이 아닌 탐색 기능에 집중한다.
