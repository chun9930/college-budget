# college-budget-web

대학생이 지금 소비를 해도 되는지 빠르게 판단할 수 있도록 돕는 React + Vite 기반 지출 관리 웹 애플리케이션이다.

## 1. 프로젝트 목표

- 오늘 사용 가능 금액을 즉시 확인한다.
- 소비 기록보다 소비 판단을 우선한다.
- 홈은 현재 소비 상태와 행동 가이드를 가장 먼저 보여주는 즉시 판단 화면이다.
- 홈의 판단 문장은 가장 크게 보이고, 상태 배지는 그 아래 보조 정보로 둔다.
- 홈의 종소리 버튼은 실제 푸시 알림이 아니라 앱 내부 판단/경고 기록을 보여준다.
- 로그인하지 않은 사용자는 로그인/회원가입 화면만 볼 수 있다.
- Header와 BottomNav는 로그인한 상태에서만 표시한다.
- 로그인 화면에는 mock auth 테스트 계정 안내가 표시된다.
- 예산 설정은 입력값과 저장값을 분리하고, 저장하기를 눌러야만 적용된다.
- 예산 설정 페이지는 숫자 입력을 실시간 검증하고 저장 전에는 계산값을 바꾸지 않는다.
- 저장이 성공하면 각 페이지는 공통 토스트로 결과를 알려 준다.
- 지출 기록은 입력 중인 금액의 소비 가능 여부를 미리 보여주는 입력 판단 화면이다.
- 최근 기록 기반 자동 입력 보조로 빠른 지출 입력을 지원한다.
- 소비 상태와 경고를 직관적으로 전달한다.
- 복잡한 분석이 아닌 이해 가능한 요약 정보를 제공한다.

## 3. 페이지 구조

Hash Router 기준 경로는 다음과 같다.

- `/#/` -> 홈
- `/#/budget-settings` -> 예산 설정
- `/#/expense-records` -> 지출 기록
- `/#/statistics` -> 분석
- `/#/calendar` -> 캘린더
- `/#/login` -> 로그인
- `/#/signup` -> 회원가입
- `/#/my-page` -> 마이페이지

비로그인 상태에서는 `/`, `/budget-settings`, `/expense-records`, `/statistics`, `/calendar`, `/my-page` 접근 시 `/login`으로 이동한다.
로그인 상태에서는 `/login`과 `/signup`에 접근해도 홈으로 되돌아간다.

## 4. 데이터 저장 방식

모든 데이터는 `localStorage` 기반으로 관리한다.

`src/lib/storage.js`를 통해서만 접근한다.

### 사용 키

- `monthlyIncome`
- `budgetSettings`
- `budgetSettings.incomeMode`
- `budgetSettings.hourlyWage`
- `budgetSettings.workHoursPerDay`
- `budgetSettings.workDaysPerWeek`
- `budgetSettings.autoIncludeRecurringExpenses`
- `savingGoalSettings`
- `savingGoals`
- `expenseRecords`
- `alertState`
- `alertHistory`
- `notificationHistory`는 이전 버전 호환용 별칭이며, 읽을 때 `alertHistory`로 병합한다.
- `recurringExpenses`
- `recurringExpenses`는 정기지출 합계를 계산할 때만 사용한다.
- `budgetSettings.autoIncludeRecurringExpenses`가 켜져 있으면 `recurringExpenses` 합계를 `수동 고정지출`에 더해 `예산에 반영되는 고정지출`로 예산 계산에 반영한다.
- `budgetSettings.manualCarryOverEnabled`
- `budgetSettings.manualCarryOverAmount`
- `carryOverState`
- `carryOverState`는 월별 자동 이월 스냅샷을 저장한다.
- `users`
- `users`에는 `name`, `email`, `password`만 저장한다.
- `mockAccountData`
- `mockAccountData`에는 로그인한 계정별로 예산/기록/정기지출/알림 기록 스냅샷을 저장한다.
- `mockAccountData` 스냅샷은 `budgetSettings`, `savingGoalSettings`, `savingGoals`, `expenseRecords`, `recurringExpenses`, `alertState`, `alertHistory`, `carryOverState`를 포함하도록 정규화한다.
- `seedMockDataIfNeeded()`는 기존 `users`를 유지한 채 mock auth 테스트 계정과 데모 데이터를 이메일 기준으로 병합해 넣는다.
- seed 데이터는 포트폴리오/기능 확인용이며 실제 사용자 데이터가 아니다.
- seed로 생성되는 테스트 계정은 `1234@naver.com / 1234`와 `student@pingo.com / 12345`이다.
- `loginState`
- `userProfile`
- 데이터 초기화는 서비스 데이터만 비우고 `users`, `loginState`, `userProfile`은 유지한다.
- 계정별 서비스 스냅샷은 `mockAccountData`에 별도로 저장한다.
- seed 주입은 `users`와 `mockAccountData`에서 필요한 테스트 계정만 추가하고, 기존 데이터는 덮어쓰지 않는다.
- `currentDate`는 저장하지 않고 앱 실행 시점의 기준 날짜로만 사용한다.

## 5. 회원가입 / 로그인 정책

회원가입과 로그인은 실제 서버 인증이 아니라 `localStorage` 기반 mock auth로 구현한다.

목적은 실제 보안 인증이 아니라 프론트엔드 사용자 흐름을 구현하는 것이다.

### 로그인 페이지

- 이메일 입력
- 비밀번호 입력
- 로그인 버튼
- 회원가입 페이지 이동

### 회원가입 페이지

- 사용자 이름 입력
- 이메일 입력
- 비밀번호 입력
- 비밀번호 확인 입력
- 회원가입 버튼
- 로그인 페이지 이동

### 마이페이지

- 사용자 이름 표시
- 이메일 표시
- 예산 요약
- 목표 요약
- 기능 토글 상태
- 데이터 초기화
- 로그아웃

### 동작 기준

- 회원가입 시 `users`에 사용자 정보를 저장한다.
- 같은 이메일로 중복 가입할 수 없다.
- 로그인 시 `users`에서 이메일과 비밀번호가 일치하는지 확인한다.
- 이메일 형식이 아니면 가입/로그인을 진행하지 않고 인라인 에러를 보여 준다.
- 로그인 성공 시 `loginState`와 `userProfile`을 저장한다.
- 로그인 성공 후에는 홈(`/`)으로 이동한다.
- 로그아웃 시 `loginState`와 `userProfile`을 제거한다.
- 로그아웃 후에는 로그인(`/login`)으로 이동한다.
- 비밀번호 찾기, 이메일 인증, 서버 인증, 토큰 인증은 구현하지 않는다.
- 회원가입 성공 후에는 자동 로그인하지 않고 로그인 페이지로 이동한다.
- 로그인/회원가입은 mock auth이며, seed 계정으로도 바로 로그인할 수 있다.
- 로그인 화면의 테스트 계정 안내 박스에 보이는 계정은 실제 `users`에 저장되어 로그인 가능해야 한다.
- 테스트 계정은 포트폴리오/기능 확인용이며 실제 보안 인증 정보가 아니다.
- 예산 설정 화면은 `예산 계산 설정`과 `목표 계획 목록` 두 탭으로 나뉜다.
- `예산 계산 설정` 탭에는 월수입, 월 저축 계획, 수동 고정지출, 비상금, 이월, 기능 토글, 정기지출 자동 반영 설정이 들어간다.
- `목표 계획 목록` 탭에는 `savingGoals` 배열로 여러 목표를 카드 목록으로 관리하고, 기존 `savingGoalSettings`는 예산 계산용 단일 목표로 유지한다.
- 예산 설정 값은 저장 버튼을 눌러야만 실제 예산 계산에 반영된다.
- 예산 설정 입력 중에는 저장되지 않은 변경사항을 구분해서 보여 준다.
- 금액 입력값이 비어 있으면 저장 시 0으로 안전 처리하고, 음수/문자는 인라인 오류로 저장을 막는다.

## 6. 핵심 계산 로직

### 하루 예산

하루 예산 =

`(월 수입 또는 수동 예산 + 이월 금액 - 목표 저축 - 비상금 - 고정지출 - 누적 지출) / 남은 일수`

- 남은 일수는 오늘을 포함한다.
- 수동 예산 입력값이 있으면 수동 예산을 우선한다.
- 날짜 기준은 앱에서 내려주는 `currentDate`와 공용 `getToday()` 기준으로 통일한다.

### 목표 설정 반영

- 목표 설정 OFF: 목표 금액 계산에서 제외한다.
- 목표 설정 ON: 목표 기간 기준 필요 저축 금액을 계산한 뒤 예산에서 차감한다.

### 기간별 저축 계산

필요 저축 =

`(목표 금액 - 현재 저축) / 남은 기간`

출력:

- 하루
- 주
- 월

### 이월 기능

- ON: 자동 계산된 이월 금액을 예산에 추가한다.
- OFF: 이월하지 않는다.
- 수동 이월 금액 입력을 켠 경우에만 직접 이월 금액을 입력할 수 있다.
- 수동 이월 값이 비어 있거나 0 이하이면 수동 이월은 저장 시 OFF로 정리된다.

### 경고 상태 기준

- 안전: 0~69%
- 주의: 70~89%
- 위험: 90~99%
- 초과: 100% 이상

### 초과 처리

- 예산은 변경하지 않는다.
- 알림 상태만 다시 계산한다.

## 7. 지출 기록

### 지출 기록

- 지출 기록 페이지는 `일반 지출 기록`과 `정기지출 관리` 두 내부 페이지로 나뉜다.
- 일반 지출 기록 안내는 식비, 카페, 교통비처럼 실제 생활 중 발생한 소비를 기록하는 곳이라고 설명한다.
- 자동 반영된 정기지출은 기록에는 남지만, 예산 계산에서는 중복 제외된다는 안내를 보여 준다.
- 일반 지출 기록 내부 페이지는 데스크톱에서 지출 입력 폼, 일반 지출 기록 목록, 최근 기록으로 빠른 입력을 3열로 보여 준다.
- 금액
- 카테고리 버튼 그룹
- 결제수단 버튼 그룹
- 일반 지출 카테고리와 정기지출 카테고리는 서로 분리해서 사용한다.
- 기존 저장 category 값이 새 목록에 없으면 `기존: {category}` 옵션으로 표시한다.
- 금액 입력 중에는 저장 전 소비 가능 여부를 보여준다.
- 일반 지출과 정기지출의 금액 입력은 실시간으로 검증하고, 빈 값/0/음수/숫자 아님에 대해 인라인 에러를 보여준다.
- 예산 설정이 없으면 예산 설정이 필요하다는 안내를 보여준다.

### 고급 옵션

- 메모는 기본 상태에서는 숨기고 고급 옵션 펼치기 안에 둔다.
- 고급 옵션 토글은 `고급 옵션 펼치기` / `고급 옵션 접기` 두 상태로 표시한다.

### 최근 기록 기반 자동 입력 보조

- 최근 기록 10개를 빠른 입력 영역으로 보여 준다.
- 최근 기록을 누르면 금액, 카테고리, 결제수단이 입력 폼에 채워진다.
- 캘린더에서 선택한 날짜는 지출 저장 날짜의 기본값으로 사용한다.
- 저장이 성공하면 공통 토스트로 결과를 보여 준다.

### 일반 지출 관리

- 일반 지출 기록 목록에서 수정할 수 있다.
- 일반 지출 기록 목록에서 삭제할 수 있다.
- 수정 중에는 지출 기록 수정 중 상태를 표시한다.
- 수정 저장 시 기존 지출 기록의 id를 유지한다.
- 일반 지출은 고급 옵션에서 지출 유형을 선택하지 않고 항상 일반으로 저장한다.
- 일반 지출의 저장/수정/삭제 성공 시 토스트를 보여 준다.

### 정기지출 관리

- 정기지출 항목을 수정할 수 있다.
- 정기지출 항목을 삭제할 수 있다.
- 정기지출 카테고리와 결제수단은 버튼 그룹으로 선택한다.
- 정기지출 결제일은 날짜 입력으로 선택하며, 새 등록 시 기본값은 오늘 날짜다.
- 정기지출 결제일은 저장 시 일(day)만 보관한다.
- 수정 중에는 현재 항목의 이름과 금액을 표시한다.
- 정기지출 안내는 월세, 통신비, 구독료처럼 반복되는 고정비라고 설명하고, 수동 고정지출과는 별도로 저장된다는 점을 안내한다.
- 정기지출의 저장/수정/삭제 성공 시 토스트를 보여 준다.

## 8. 분석 리포트

- 이번 달 총 지출을 가장 먼저 보여준다.
- 분석 화면에서 월을 선택하면 해당 월 기준으로 통계를 다시 계산한다.
- 기본값은 currentDate가 속한 이번 달이다.
- 월을 바꾸면 카테고리 비중, 전월 대비 변화, 소비 패턴 피드백, 대학생 절약 팁이 함께 갱신된다.
- 카테고리별 지출 비중 카드는 상단부터 내용이 시작되도록 배치하고, 피드백 카드와 균형 있게 보이도록 정리했다.
- 일별 지출 막대와 카테고리 비중을 순서대로 보여준다.
- 가장 많이 쓴 항목을 한 줄로 보여준다.
- 카테고리별 비중, 전월 대비 변화, 과소비 가능성 피드백을 함께 보여 준다.
- 대학생 절약 팁 섹션에서 카테고리별 실천 행동을 함께 보여 준다.
- 대학생 절약 팁은 카테고리별 고정 데이터 기반으로 보여 주며, 현재 소비 패턴과 연결된 실천 행동을 먼저 제안한다.
- 자동 반영된 정기지출은 기록에는 포함되지만 예산 계산에서는 중복 제외된다는 안내를 표시한다.
- 경고나 소비 가능 여부 판단은 넣지 않고, 분석 피드백 카드로만 보여 준다.

## 9. UI 구조 원칙

- 한 화면에 하나의 목적만 둔다.
- 정보는 요약 -> 핵심 -> 상세 순서로 배치한다.
- 색상 + 텍스트로 상태를 전달한다.
- 버튼 최소 높이 48px 유지
- 텍스트 최소 16px 이상
- 모바일 우선 설계
- Toss + Banksalad 하이브리드 톤을 유지한다.
- 카드 구조는 최소화하고, 핵심 숫자를 먼저 보여준다.
- 홈은 판단 결과를 가장 크게 보여준다.
- 홈의 판단 문장은 최상단 중앙에 크게 보이고, 경고 배너는 접기/펼치기로 숨길 수 있다.
- 홈의 경고 배너는 접기/펼치기 가능한 로컬 상태로만 숨기고 새로고침 시 다시 보이게 한다.
- 홈의 빠른 이동 버튼은 웹 데스크톱에서 12px 간격, 40~48px 높이 기준으로 보이게 한다.
- 홈의 빠른 이동 영역에는 지출 입력, 예산 설정, 분석 보러가기 버튼을 둔다.
- 홈 상단 우측의 종소리 버튼은 최근 판단/경고 메시지 기록을 보여준다.
- 로그인/회원가입 화면에서는 Header와 BottomNav를 숨긴다.
- 로그인 화면에는 테스트 계정 안내 박스를 둔다.
- 마이페이지의 데이터 초기화는 로그인 상태와 사용자 정보를 유지한 채 서비스 데이터만 비운다.
- 지출 기록은 입력 중인 금액 기준 판단 문구와 상태를 금액 입력 아래에 보조로 보여준다.
- 예산 설정의 `수동 고정지출`은 이번 달 예산에서 미리 빼둘 계획 지출로 입력하며, `정기지출 합계 불러오기` 버튼은 표시하지 않는다.
- 예산 설정 화면에는 이번 달 예산 초과 여부를 별도로 보여 주어 홈의 오늘 예산 판단과 분리한다.
- `정기지출`은 `recurringExpenses` 목록으로만 관리하고, 수동 고정지출에 복사해서 넣지 않는다.
- 정기지출 자동 반영이 켜져 있으면 저장된 `수동 고정지출`과 `정기지출 합계`를 함께 사용하고, 중복 차감되지 않도록 `sourceRecurringId`가 있는 기록은 예산 계산용 월 지출 합계에서 제외한다.
- 예산 설정 화면에는 `수동 고정지출`, `정기지출 합계`, `예산에 반영되는 고정지출`만 사용자용 라벨로 보여 준다.
- 통계 화면에서는 자동 반영된 정기지출이 기록에는 포함되지만 예산 계산에서는 중복 제외된다는 안내 문구를 보여준다.
- 자동 이월은 월별 스냅샷으로 저장해 같은 달에는 다시 계산하지 않고, 이전 달 기준 결과를 재사용한다.
- 월 수입은 `직접 입력`과 `근무 조건 계산` 중 하나를 선택해 저장할 수 있다.
- 근무 조건 계산은 `시급 × 하루 근무시간 × 주 근무일수 × 4.345`로 예상 월 수입을 만든 뒤 `monthlyIncome`에 적용한다.
- `BudgetSettings`와 `Home`은 같은 `monthlyIncome`, `totalFixedExpense`, `dailyBudget` 기준을 사용한다.
- `App`은 `currentDate`를 기준으로 하루/이번 달/남은 일수/자동 이월 계산을 맞춘다.
- 예산 설정은 `예산 설정`, `예산 계산 설정`, `목표 계획 목록`, `월 저축 계획`, `수입 입력 방식`, `수동 고정지출`, `이월 금액` 같은 한국어 라벨과 저장 안내 문구로 입력과 적용을 분리해 보여준다.
- 금액 입력은 실시간 검증과 인라인 에러 메시지로 처리한다.
- 분석의 일별 지출 막대는 막대 아래에 일별 총 지출 금액을 검정색으로 보여 주고, 그 아래에 회색 `N일 총 지출` 라벨을 함께 보여 준다.
- 캘린더는 `달력 보기` 버튼과 `이번달 지출` 요약을 먼저 두고, 그 아래에 접기/펼치기 가능한 달력 보조 영역과 월별 지출 내역을 배치한다.
- 캘린더의 날짜를 누르면 선택 날짜가 지출 기록 작성 기준으로 전달되고, 지출 항목을 누르면 해당 기록의 수정 화면으로 이동할 수 있다.
- 캘린더의 금액 축약 표시는 한국식 `만` 단위를 사용한다.
- 기능 토글은 스위치 행 형태로 표시한다.
- 지출 기록 페이지는 일반 지출 기록과 정기지출 관리를 분리한 내부 페이지 구조로 보여 준다.
- 일반 지출 기록 내부 페이지는 데스크톱에서 3열, 태블릿에서는 2열 또는 1열, 모바일에서는 1열로 자연스럽게 배치한다.
- 저장 성공 피드백은 공통 토스트로 보여 준다.
